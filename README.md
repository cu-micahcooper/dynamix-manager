# dynamix-manager

Notebook-first executive analytics workspace for Cedarville University's TeamDynamix environment. The repository ingests survey and ticket data from the TeamDynamix API, stores it in a local DuckDB database, and generates HTML reports and Jupyter notebooks for analysis.

## Local Environment Bootstrap

This project requires Python 3.13+.

Run the helper script from the repo root to prepare the working environment:

```bash
./scripts/bootstrap_env.sh
```

The script:

- ensures you are working from the repository root
- creates `.venv` if it does not exist
- activates the virtual environment, upgrades `pip`, and installs the project in editable mode with dev dependencies (`.[dev]`)

A successful run leaves you with an up-to-date `.venv`, so you can activate it and run verification commands.

## `.env` expectations

The workspace keeps TeamDynamix credentials in a `.env` file (not checked in). At a minimum, the following variables must be present before notebook or CLI tasks can run:

```text
TDX_BASE_URL=<TeamDynamix API base URI>
TDX_APP_ID=<TeamDynamix application ID>
TDX_USERNAME=<TeamDynamix username>
TDX_PASSWORD=<TeamDynamix password>
```

The live code can call `dynamix_manager.config.required_env_vars()` to compute this list, so keep `.env` aligned with that helper if the list changes.

## CLI Commands

The `dynamix-manager` CLI is the primary way to refresh data and generate reports. Activate the virtual environment first, then run any of the following:

### `refresh-surveys`

Fetches the latest survey report from TeamDynamix, backfills linked ticket context, materializes the `ticket_linked_surveys` model, writes `reports/survey_health.html`, and updates `notebooks/survey_health.ipynb`.

```bash
python -m dynamix_manager.cli refresh-surveys
```

### `backfill-tickets`

Incrementally fetches ticket context for all surveyed tickets in batches. Use after `refresh-surveys` to fill in any tickets that were rate-limited or missed.

```bash
python -m dynamix_manager.cli backfill-tickets [--batch-size N] [--max-batches N]
```

Options:
- `--batch-size` — number of tickets to fetch per batch (default: 50)
- `--max-batches` — maximum number of batches to run (default: 5)

### `quality-check`

Runs ticket quality analysis for open tickets, generating interaction-level flags and writing `reports/ticket_quality.html`.

```bash
python -m dynamix_manager.cli quality-check <ticket_app_id> [--limit N]
```

Options:
- `ticket_app_id` — TeamDynamix ticketing application ID (required positional argument)
- `--limit` — maximum number of tickets to process

### `cache-days-off`

Fetches and caches planned days off and holidays from TeamDynamix for use in business-day calculations.

```bash
python -m dynamix_manager.cli cache-days-off
```

## Notebooks

Notebooks live in `notebooks/` and are generated or updated by the CLI commands. Open them with JupyterLab:

```bash
source .venv/bin/activate
jupyter lab
```

Available notebooks:

| Notebook | Description |
|---|---|
| `notebooks/survey_health.ipynb` | Survey satisfaction analysis — updated by `refresh-surveys` and `backfill-tickets` |
| `notebooks/ticket_health.ipynb` | Ticket resolution time, backlog, and team-level health metrics |
| `notebooks/ticket_quality.ipynb` | Open-ticket quality flags and interaction analysis — updated by `quality-check` |

## Generated Reports

HTML reports are written to the `reports/` directory:

| Report | Generated by |
|---|---|
| `reports/survey_health.html` | `refresh-surveys`, `backfill-tickets` |
| `reports/ticket_health.html` | ticket health pipeline |
| `reports/ticket_quality.html` | `quality-check` |

## Data Storage

All ingested data is stored in `data/analytics.duckdb` (a local DuckDB file, not checked in). Key tables:

- `survey_responses` — raw survey report rows from TeamDynamix
- `tickets` — ticket context records linked to surveys
- `ticket_linked_surveys` — modeled join of surveys and tickets
- `days_off` — cached holidays and planned days off
- `ticket_quality_tickets`, `ticket_quality_interactions`, `ticket_quality_flags` — open-ticket quality analysis

## Verification

Activate the virtual environment and run each command after bootstrapping:

```bash
source .venv/bin/activate
python -m pytest
python -m ruff check src/ tests/
jupyter lab --version
```

If any verification step fails, re-run `./scripts/bootstrap_env.sh` to ensure dependencies sync and rerun the failing command.
